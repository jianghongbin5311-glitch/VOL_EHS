# AGENT_RULES.md

> 本文件用于定义 **AI Agent 在本项目中必须遵循的工程级规则**。
> 本项目属于【已有代码的老项目 / 生产项目】，Agent 必须采取保守、克制的工程策略。

---

## 零、项目架构与技术栈上下文 (Context)

**Agent 在进行任何代码分析前，必须先理解本项目的技术背景：**

1.  **总体架构**：基于 **VOL (VolCore)** 框架的前后端分离项目。
    - **前端**：Vue.js + Element UI
    - **后端**：.NET Core + EF Core

2.  **代码定位指南（哪里才是业务逻辑？）**：
    - **后端核心**：业务逻辑主要集中在 `Services/` 和 `Controllers/`。请勿随意修改 `Vol.Core/` 等底层框架代码。
    - **前端核心**：页面逻辑在 `src/views/`，扩展组件在 `src/components/`。请勿读取 `dist/` 或 `node_modules/`。

3.  **数据库交互**：
    - 本项目使用 EF Core / Dapper 混合模式。在修改数据查询时，优先检查 `Repositories/` 目录下的现有实现。

## 二、不可协商规则（Non-Negotiable Rules）

以下规则 **必须严格遵守，不得例外**：

### 2.1 禁止重构无关代码
- 仅允许修改与当前任务 **直接相关** 的代码
- 严禁行为：
  - 因“代码更优雅”
  - 因“顺手优化”
  - 因“最佳实践”
  而扩大修改范围

### 2.2 保持现有行为不变（除非明确要求）
- 不得改变：既有功能逻辑、默认配置、返回结构、边界处理行为
- **除非用户明确要求改变行为**

### 2.3 禁止破坏公共接口
- 不修改已有 API 的：路径、入参、出参、语义含义
- 如确需变更：必须先提出方案，等待人工明确确认

### 2.4 禁止引入新技术栈或依赖
- 不新增：框架、第三方库、中间件
- 不升级：核心依赖、基础运行环境

---

## 三、修改范围与方式约束（Minimal Change）

### 3.1 修改范围
- 仅允许修改：任务明确指定的文件
- 如需新增文件：必须说明原因，不得隐式引入连锁修改

### 3.2 修改方式
- 严格遵循 **最小修改原则（Minimal Change）**
- 保持：原有代码风格、原有命名规范、原有目录层级
- 若存在不确定：**优先模仿现有代码**，不得自行发挥

---

## 四、代码风格与工程规范

Agent 必须遵循项目内已有约定，包括但不限于：
- 命名方式（类 / 方法 / 变量）
- 目录结构
- 注释风格
- 异常处理方式
- 日志规范

> 当规则未明确规定时，**以现有代码为最高准则**。

---

## 五、性能与分析范围约束（用于加快响应）

### 5.1 强制忽略区域
Agent 在思考路径中必须主动排除以下目录（即使物理文件存在）：
- `node_modules`, `bin`, `obj`, `dist`

### 5.2 快速优先原则
- 优先基于用户指定文件和当前上下文。
- 若信息不足：**先提问澄清**，不得盲目扩大扫描范围（例如禁止使用“扫描整个项目”的操作）。

---

## 六、任务执行流程约束（强制）

在实现任何代码修改前，Agent **必须遵循以下流程**：

1. **Context First**：先阅读 `Startup.cs` (后端) 或 `package.json` (前端) 确认版本和依赖。
2. **Design First**：使用中文说明实现思路，列出涉及文件和潜在风险。
3. **Approve First**：经人工确认后，再逐步实现。

---

## 七、修改后的自检与验证（强制）

**每一次代码修改完成后，Agent 必须主动自检并说明：**
1. 是否影响其他功能（是 / 否 / 不确定）
2. 是否影响公共接口（是 / 否）
3. 已验证的内容（正常流程、关键边界）
4. 潜在风险与未覆盖点

> 若无法确认影响范围，必须如实说明，不得假设“应该没问题”。

---

## 八、不确定性与风险处理

当出现以下情况时，Agent **必须暂停执行并询问**：
- 需求存在歧义
- 规则之间存在冲突
- 修改可能影响核心业务链路
- 需要突破本文件约束
